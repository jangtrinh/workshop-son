2.1 — Rules Files

Từ Gem → Rules File

Gem = customize AI trong app chat. Rules File = customize AI trong code editor, áp dụng cho toàn bộ project, version control được qua Git. Sức mạnh lớn hơn nhiều: text + code + file access + terminal.

So sánh Gem vs Rules File

Gem (Level 1)Rules File (Level 2)
Nền tảngGemini App (web/mobile)Code editor (Cursor, Antigravity)
FormatTextarea plain textMarkdown file → Git version control
Scope1 conversation threadToàn bộ project / workspace
Khả năngText generationText + code + file + terminal
Chia sẻLink/copyGit repository

Cách hoạt động

Bạn mở project trong code editor
        │
        ▼
Editor tìm rules file (GEMINI.md / .cursorrules / CLAUDE.md)
        │
        ▼
Nội dung rules được INJECT vào mọi AI request
        │
        ▼
AI trả lời dựa trên: [Model] + [Rules] + [Files] + [Prompt]

Template: Rules File cơ bản

# Project Rules

## About This Project
[Mô tả ngắn project, tech stack, mục tiêu]

## Code Style
- Use TypeScript strict mode
- Prefer functional components
- Max function length: 20 lines
- Always handle errors explicitly

## Naming Convention
- Files: kebab-case (user-profile.tsx)
- Components: PascalCase (UserProfile)
- Functions: camelCase (getUserProfile)

## Architecture
- Pages in /app
- Components in /components
- Utilities in /lib

## Constraints
- DO NOT create files outside the project structure
- DO NOT install new dependencies without asking
- Always check existing utilities before creating new ones

2.2 — Workflows

Tại Sao

Workflow = chuỗi bước có thứ tự mà AI thực hiện khi trigger bằng slash command. Prompt đơn = 1 bước. Gem/Rules = lặp lại nhưng 1 bước. Workflow = nhiều bước, có thứ tự, multi-file output.

Cấu trúc Workflow File

---
description: Tóm tắt workflow làm gì (1 dòng)
---

# /workflow-name — Tiêu đề

## Bước 1: [Tên bước]
[Hướng dẫn chi tiết]

## Bước 2: [Tên bước]
[Hướng dẫn chi tiết]

Ví dụ thực tế: /ship

---
description: Automated ship pipeline — merge, test, commit, push
---

# /ship — Ship Code Pipeline

## Step 1: Merge Latest
Pull latest from main → avoid conflicts.

## Step 2: Run Tests
Full test suite. If fail → STOP and report.

## Step 3: Lint & Format
Run linter. Auto-fix where possible.

## Step 4: Version Bump
fix → patch | feat → minor | breaking → major

## Step 5: Commit
Conventional commit with proper scope.

## Step 6: Push & Create PR
Push branch + create pull request.
// turbo annotation

Đặt // turbo trước step để AI tự chạy không hỏi permission. Dùng cho safe steps (lint, test). // turbo-all = auto-run toàn bộ workflow.

2.3 — Khi Nào KHÔNG Dùng Agent

Over-Automation Anti-Patterns

Anti-PatternVấn đềGiải pháp
"Automate Everything"Setup > tiết kiệmRule of 3: chỉ automate task lặp >3/tuần
"AI viết, tôi paste"Technical debtLuôn review + hiểu output
"1 mega prompt"AI overload1 task = 1 prompt. Workflow cho multi-step
"AI thay thế thinking"Bad decisionsAI assist ≠ AI decide
"Context dump"Noise > signalSelective context, chỉ file/fn liên quan
"Trust first output"HallucinationVerify facts, test code

Framework: Nên Automate Không?

                    EFFORT TO AUTOMATE
                Low ◄──────────────► High
             ┌──────────┬──────────────┐
   High      │   DO IT │   EVALUATE │
FREQUENCY    │  No       │  ROI calc    │
             │  brainer  │  needed      │
             ├──────────┼──────────────┤
   Low       │   MAYBE │   DON'T    │
             │  If easy  │  waste of    │
             │  why not  │  time        │
             └──────────┴──────────────┘

KHÔNG dùng khi

Task quá đơn giản (< 2 phút tự làm)
Cần 100% chính xác (legal, financial, medical)
Không có cách verify output
Cần empathy / emotional intelligence thật
Security-sensitive (secrets, credentials)

NÊN dùng khi

Task lặp với pattern rõ ràng
Cần explore nhiều options nhanh
Boilerplate code / documentation
Refactoring theo quy chuẩn có sẵn
Research & summarization
